0.3 — Cliente Axios (src/lib/axios.ts)

Una instancia de Axios preconfigurada con la URL base y el API key en el header. Así todos los hooks de TanStack Query usan la misma configuración.

  baseURL: import.meta.env.VITE_COINGECKO_API_URL,

📖
Línea por línea:

1.1 — Tipos (src/types/index.ts)

Un archivo con todas las interfaces que usaremos en toda la app. TypeScript las resuelve automáticamente sin necesidad de importar (porque index.ts es un barrel, pero aquí sí importaremos porque queda explícito).

export interface Coin {
  id: string;  // "bitcoin"
  symbol: string;  // "btc"
  name: string;  // "Bitcoin"
  image: string;  // URL del logo
  current_price: number;
  market_cap: number;
  market_cap_rank: number;
  total_volume: number;
  high_24h: number | null;
  low_24h: number | null;
  price_change_percentage_24h: number | null;
  circulating_supply: number;
  total_supply: number | null;
  max_supply: number | null;
  last_updated: string;
}

export interface CoinHistoryPoint {
  timestamp: number;  // UNIX ms
  price: number;
}

export interface AuthUser {
  id: string;
  email: string;
  name: string;
  avatar_url: string;
}

export interface LoginCredentials {
  email: string;
  password: string;
}
📖
¿Por qué estos tipos? Son los campos que devuelve CoinGecko exactamente. Cada | null significa que "puede venir null desde la API". TypeScript te obliga a manejar ese caso — no puedes asumir que siempre hay un valor.
¿Por qué no tipamos el histórico con [number, number][]? Porque CoinHistoryPoint es más semántico: en vez de data[0][0] escribes point.timestamp.

1.2 — authStore (src/stores/authStore.ts)

Store de Zustand que reemplaza a UserContext. Maneja token y sesión de usuario, preparado para swap a Supabase después.

import { create } from "zustand";
import type { AuthUser } from "../types";

interface AuthState {
  user: AuthUser | null;
  token: string | null;
  setSession: (user: AuthUser, token: string) => void;
  logout: () => void;
}

export const useAuthStore = create<AuthState>((set) => ({
  user: null,
  token: localStorage.getItem("tokenCriptoBro"),
  setSession: (user, token) => {
    localStorage.setItem("tokenCriptoBro", token);
    set({ user, token });
  },
  logout: () => {
    localStorage.removeItem("tokenCriptoBro");
    set({ user: null, token: null });
  },
}));
📖
Línea por línea:
create<AuthState>((set) => ({})) — Zustand usa create con un callback que recibe set para actualizar estado. El genérico <AuthState> tipa el store.
token: localStorage.getItem(...) — al crearse el store, lee el token guardado (si existe). Así al recargar la página, el usuario no pierde la sesión.
set({ user, token }) — Zustand reemplaza (merge superficial) el estado con el objeto que le pases. Equivalente a useState pero global.
¿Por qué Zustand y no Context? Porque Zustand no provoca re-renders en componentes que no escuchan ese slice de estado. Context rerenderiza a todos los hijos aunque no usen el valor.

1.3 — main.tsx (reescribir)

El entry point. Aquí montamos QueryClientProvider y BrowserRouter para que toda la app tenga acceso a TanStack Query y React Router.

import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { BrowserRouter } from "react-router-dom";
import App from "./App";
import "./App.css";

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 60_000,  // 1 min antes de refetch
      refetchOnWindowFocus: false,  // no recargar al volver a la pestaña
    },
  },
});

createRoot(document.getElementById("root")!).render(
  <StrictMode>
    <QueryClientProvider client={queryClient}>
      <BrowserRouter>
        <App />
      </BrowserRouter>
    </QueryClientProvider>
  </StrictMode>,
});
📖
Línea por línea:
new QueryClient({defaultOptions: {queries: {staleTime, refetchOnWindowFocus}}}) — configura las queries globales. staleTime: 60_000 significa: "los datos se consideran frescos durante 1 minuto". Si un componente pide los mismos datos dentro del minuto, TanStack Query devuelve la caché sin hacer fetch.
document.getElementById("root")! — el ! le dice a TS: "confía en mí, esto no es null". Sin el !, TS se quejaría porque getElementById puede retornar null.
¿Por qué movemos BrowserRouter a main.tsx? Porque App.tsx solo debe tener rutas, no providers. Así AppLayout puede usar <Outlet /> sin problemas.
2 TanStack Query — hooks de CoinGecko

Tres hooks (KISS: uno por endpoint) + un hook de login. Cachean automáticamente, manejan carga/error, y eliminan la necesidad de useData.js.

2.1 — src/api/coins.ts (lista)

Hook para obtener las 100 criptos principales.

import { useQuery } from "@tanstack/react-query";
import { coingeckoApi } from "../lib/axios";
import type { Coin } from "../types";

const fetchCoins = async (): Promise<Coin[]> => {
  const { data } = await coingeckoApi.get<Coin[]>("/coins/markets", {
    params: { vs_currency: "usd", order: "market_cap_desc", per_page: 100 },
  });
  return data;
};

export const useCoins = () =>
  useQuery({ queryKey: ["coins"], queryFn: fetchCoins });
📖
Línea por línea:
coingeckoApi.get<Coin[]>(url, {params}) — el genérico <Coin[]> tipa el data que devuelve Axios. Sin esto, data sería any.
useQuery({queryKey: ["coins"], queryFn: fetchCoins}) — queryKey es un identificador único para esta query. Cuando otro componente pida ["coins"], TanStack Query devuelve la misma data cacheada. queryFn es la función que hace el fetch.
¿Qué devuelve useQuery? { data, isLoading, error, refetch, ... }. data será Coin[] | undefined, isLoading es boolean, error es Error | null.

2.2 — src/api/coinDetail.ts (detalle individual)

Reusa el mismo endpoint /coins/markets pero filtra por ?ids=bitcoin.

import { useQuery } from "@tanstack/react-query";
import { coingeckoApi } from "../lib/axios";
import type { Coin } from "../types";

const fetchCoin = async (id: string): Promise<Coin> => {
  const { data } = await coingeckoApi.get<Coin[]>("/coins/markets", {
    params: { vs_currency: "usd", ids: id },
  });
  return data[0];
};

export const useCoinDetail = (id: string | undefined) =>
  useQuery({
    queryKey: ["coin", id],
    queryFn: () => fetchCoin(id!),
    enabled: !!id,  // solo ejecuta si id no es undefined
  });
📖
Línea por línea:
enabled: !!id — !!id convierte string | undefined a boolean. Si id es undefined (cuando el parámetro de ruta no se ha cargado), la query no se ejecuta. Esto evita un fetch con ?ids=undefined.
queryKey: ["coin", id] — al incluir id en la key, cada cripto tiene su propia caché. ["coin", "bitcoin"] es distinto de ["coin", "ethereum"].
¿Por qué id! en queryFn? TypeScript no sabe que enabled: !!id garantiza que id no es undefined dentro de queryFn. El ! es un "non-null assertion": le decimos a TS "confía, no va a ser undefined aquí".

2.3 — src/api/coinHistory.ts (histórico de precios)

Obtiene los precios históricos para la gráfica. La Demo API solo permite hasta 365 días.

import { useQuery } from "@tanstack/react-query";
import { coingeckoApi } from "../lib/axios";
import type { CoinHistoryPoint } from "../types";

interface ChartResponse {
  prices: [number, number][];  // [timestamp_ms, price_usd][]
}

const fetchHistory = async (id: string, days: string): Promise<CoinHistoryPoint[]> => {
  const { data } = await coingeckoApi.get<ChartResponse>(
    `/coins/${id}/market_chart`,
    { params: { vs_currency: "usd", days } }
  );
  return data.prices.map(([ts, price]) => ({ timestamp: ts, price }));
};

export const useCoinHistory = (id: string | undefined, days: string) =>
  useQuery({
    queryKey: ["coin-history", id, days],
    queryFn: () => fetchHistory(id!, days),
    enabled: !!id,
  });
📖
Línea por línea:
interface ChartResponse { prices: [number, number][] } — tipamos la respuesta cruda de CoinGecko. [number, number][] significa "array de tuplas de 2 números".
data.prices.map(([ts, price]) => ({timestamp: ts, price})) — transformamos la tupla en un objeto con nombre. En TS, [ts, price] es destructuring de array: el primer elemento va a ts, el segundo a price.
¿Porqué "365 días"? CoinGecko Demo API solo permite histórico hasta 365 días atrás. Con plan Pro obtienes datos desde 2010.

2.4 — src/api/auth.ts (login mutation)

Hook de mutación para login con reqres.in (temporal). Cuando migres a Supabase, solo cambiarás el cuerpo de la mutation.

import { useMutation } from "@tanstack/react-query";
import axios from "axios";
import { useAuthStore } from "../stores/authStore";
import type { LoginCredentials } from "../types";

interface LoginResponse {
  token: string;
}

export const useLoginMutation = () => {
  const setSession = useAuthStore((s) => s.setSession);

  return useMutation({
    mutationFn: async (credentials: LoginCredentials) => {
      const { data } = await axios.post<LoginResponse>(
        "https://reqres.in/api/login", credentials
      );
      return data.token;
    },
    onSuccess: (token) => {
      setSession({ id: "", email: "", name: "", avatar_url: "" }, token);
    },
  });
};
📖
Línea por línea:
useMutation — a diferencia de useQuery (que lee datos), useMutation es para operaciones que cambian datos en el servidor (POST, PUT, DELETE).
mutationFn — la función que hace la petición POST. Recibe LoginCredentials (desestructurado de useMutation).
onSuccess: (token) => { setSession(...) } — callback que se ejecuta cuando la mutation termina exitosamente. Aquí actualizamos el store de Zustand.
useAuthStore((s) => s.setSession) — suscripción selectiva a Zustand. Solo extraemos setSession, no toda la store. Así este hook no se re-renderiza cuando cambia el token.

4.7 — Login.tsx (corregir errores TS + usar mutation)

Error actual: useState<string | null>(null) y luego setUser({...user, email}) → TypeScript se queja porque no puedes hacer spread de string.

Solución: usar un objeto como estado {email: "", password: ""} y la mutation en vez de axios directo.

import { useState } from "react";
import { Navigate } from "react-router-dom";
import { useAuthStore } from "../stores/authStore";
import { useLoginMutation } from "../api/auth";
import "../css/login.css";

const Login = () => {
  const token = useAuthStore((s) => s.token);
  const { mutate, isPending, error } = useLoginMutation();
  const [credentials, setCredentials] = useState({
    email: "",
    password: "",
  });  // ✅ objeto, no string | null

  if (token) return <Navigate to="/" />;

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault();
    mutate(credentials);  // ✅ TanStack Query maneja carga/error
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={credentials.email}
        onChange={(e) => setCredentials({...credentials, email: e.target.value})}
      />
      {/* ...similar para password */}
      <button type="submit" disabled={isPending}>
        {isPending ? "Cargando..." : "Ingresar"}
      </button>
    </form>
  );
};
