1 Infraestructura: Supabase + CoinGecko

pnpm add @supabase/supabase-js

1.2 — Configurar Supabase (dashboard)

  1. Crear proyecto en supabase.com
  2. Authentication → Providers → habilitar Google, Facebook, GitHub y Email
  3. URL Configuration → Site URL http://localhost:5173, Redirect URLs http://localhost:5173/**

    VITE_SUPABASE_URL=https://xxxx.supabase.co
    VITE_SUPABASE_PUBLISHABLE_KEY=sb_publishable_xxxx

1.3 — SQL: tablas profiles y favorites

Ejecutar en SQL Editor del dashboard de Supabase:

create table public.profiles (
  id uuid primary key references auth.users on delete cascade,
  full_name text,
  avatar_url text,
  updated_at timestamptz default now()
);

create table public.favorites (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users on delete cascade,
  coin_id text not null,
  coin_name text not null,
  coin_symbol text not null,
  created_at timestamptz default now(),
  unique (user_id, coin_id)
);

Activar RLS y crear las 6 policies (3 para cada tabla). El trigger handle_new_user crea un perfil automáticamente al registrarse.

1.4 — supabaseClient.ts y tipos

import { createClient } from "@supabase/supabase-js";

const url = import.meta.env.VITE_SUPABASE_URL;
const key = import.meta.env.VITE_SUPABASE_PUBLISHABLE_KEY;

if (!url || !key) {
  throw new Error(
    "Faltan VITE_SUPABASE_URL o VITE_SUPABASE_PUBLISHABLE_KEY"
  );
}

export const supabase = createClient(url, key, {
  auth: {
    persistSession: true,
    autoRefreshToken: true,
    detectSessionInUrl: true,
  },
});

3 opciones clave:

  • detectSessionInUrl: true — procesa el #access_token del redirect de OAuth (Google/Facebook/GitHub)
  • persistSession: true — guarda el JWT en localStorage
  • autoRefreshToken: true — renueva el token antes de que expire

2 Auth con Supabase + Zustand

Reemplazar reqres.in y UserContext por Supabase Auth + store Zustand reactivo.

2.1 — authStore reactivo a Supabase

Escuchar supabase.auth.onAuthStateChange en lugar de leer localStorage manualmente. Estado: session, user, profile, loading.

⚠️
Importante — loading debe iniciarse en true: Supabase tarda un instante en restaurar la sesión al cargar (lee localStorage + valida el JWT). Hasta que llega el primer evento INITIAL_SESSION, session es null aunque el usuario sí esté logueado. El <Protected> debe esperar a loading: false antes de decidir:
const { session, loading } = useAuthStore((s) => ({
  session: s.session,
  loading: s.loading,
}));

if (loading) return <Loading />;
return session ? <>{children}</> : <Navigate to="/login" />;

Si <Protected> solo comprueba session, redirige a /login en cada reload (flash visible a /login y de vuelta).

3.4 — useFavorites (nuevo)

3 hooks contra Supabase:

  • useFavorites() → .from("favorites").select("*")
  • useAddFavorite() → .insert(...) + invalidateQueries
  • useRemoveFavorite() → .delete().eq("id", id) + invalidateQueries
⚠️
Importante — enabled reactivo a session: si usamos useAuthStore.getState().session dentro del query, es un valor estático leído una vez y la query no se re-habilita cuando el usuario hace login. Hay que leer la sesión con el hook de Zustand:
export const useFavorites = () => {
  const session = useAuthStore((s) => s.session);

  return useQuery({
    queryKey: ["favorites"],
    queryFn: async () => {
      const { data, error } = await supabase.from("favorites").select("*").order("created_at", { ascending: false });
      if (error) throw error;
      return data as Favorite[];
    },
    enabled: !!session,
  });
};
